IoC概念和原理

什么是IoC

  1. 控制反转，把对象创建和对象之间的调用过程，交给Spring进行管理（不使用传统的new）
  2. 目的：降低耦合度
  3. 入门案例就是IoC实现

IoC底层原理

主要用到三部分技术: xml解析、工厂模式、反射

通过上面的示意图来讲解。

我们现在有两个类分别是UserService和UserDao，如果我们要在UserService中调用UserDao的add()方法，最简单的就是先new一个

UserDao dao = new UserDao();

然后再调用方法

dao.add()

这种方法的问题在于耦合度太高，假设UserDao路径或是方法发生了变化，那么UserService一样要跟着变化。

为了解决这个问题，可以采用工厂模式。

我们依旧有UserService和UserDao两个类，另外再创建一个工厂类UserFactory类

class UserFactory {
    public static UserDao getDao() {
        return new UserDao();
    }
}

那么我们再调用dao的add方法就可以通过工厂类实现

class UserService {
    execute() {
        UserDao dao = UserFactory.getDao();
        dao.add();
    }
}

但这样耦合度依旧没有降到最低，所以就需要IoC进一步降低耦合

依旧通过上述示意图讲解

和入门案例相同，IoC过程的第一步先使用xml配置创建的对象

假设我们有service类和dao类，创建工厂类

class UserFactory {
  public static UserDao getDao() {
    // 先解析xml
    String classValue = class属性值;
    // 通过反射创建对象
    Class clazz = Class.forName(classValue);
    // 创建对象
    return (UserDao)clazz.newInstance();
  }
}

这样的话，假设我们UserDao的路径变了，我们只需要修改xml配置文件，其他部分都不需要改变，也就进一步降低了耦合度。

IoC接口

  1. IoC思想基于IoC容器完成，IoC容器底层就是对象工厂

  2. Spring提供IoC容器实现的两种方式：(两个接口)

    (1) BeanFactory

    (2) ApplicationContext

在入门案例中ApplicationContext接口我们用来加载xml配置文件，如果改成BeanFactory效果是相同的，区别在于：

  • BeanFactory是Spring内部的使用接口，不提供开发人员进行使用
  • ApplicationContext是BeanFactory接口的子接口，提供更多更强大的功能，一般由开发人员进行使用

还有一点区别非常重要！！！

BeanFactory加载配置文件时不会创建对象，在获取对象（使用）才去创建对象
ApplicationContext在加载配置文件时就会把在配置文件中的对象创建

  1. ApplicationContext接口的实现类

    (1) FileSystemXmlApplicationContext

    (2) ClassPathXmlApplicationContext

如果用FileSystemXmlApplicationContext，那么参数要写绝对路径，而使用ClassPathXmlApplicationContext，参数写项目内的文件路径
